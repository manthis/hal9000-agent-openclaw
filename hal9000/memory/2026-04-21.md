## Bishop and Jarvis exec hardening
- Jarvis was successfully hardened by switching to agent-level exec approvals with `security: allowlist`, `ask: on-miss`, `askFallback: deny`, plus Telegram exec approvals enabled for `jarvis`.
- Jarvis needed script refactors because inline forms like `python3 -c ...` and heredoc Python blocked clean allowlisting. `jarvis_status` and `invitations` were converted to file-backed scripts.
- Bishop audit found it structurally cleaner than old Jarvis, but politically too permissive: host exec effectively `security: full`, `ask: off`, no agent override, and Telegram exec approvals filtered to `jarvis` only.
- Bishop was then hardened like Jarvis: agent policy moved to `security: allowlist`, `ask: on-miss`, `askFallback: deny`, with allowlist for its normal Python/OpenClaw scripts and working files.
- If Bishop should request approvals in Telegram DMs for non-allowlisted exec, `channels.telegram.execApprovals.agentFilter` still needs `bishop` added; at audit time it only included `jarvis`.
- Bishop durci comme Jarvis avec approvals Telegram actifs et policy `security: allowlist`, `ask: on-miss`, `askFallback: deny`.
- Correctif de fond Bishop appliqué et poussé, `scripts/bishop_status.py` et `scripts/bishop_watch.py` lisent l’état réel des commentaires via l’API Unipile plutôt que seulement `memory/queue.json`; commit `ed29ad1` (`fix: use live comment state in bishop status`).
- `/likes` Bishop a besoin d’être allowlisté côté fichiers de skill, notamment `skills/bishop-likes/SKILL.md`, en plus des scripts et JSON déjà présents.
